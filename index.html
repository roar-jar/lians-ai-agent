AI 에이전트 만들기: 답변을 잘 받는 법에서, 일을 잘 맡기는 법으로 넘어갑니다.

오늘의 목표

이번 시간이 끝나면 할 수 있어야 하는 것

  • AI가 답을 만드는 기본 원리를 한 문장으로 설명한다.
  • 모델이 커지거나 길게 생각해도 결과가 나빠질 수 있는 이유를 안다.
  • AI에게 잘 맡길 수 있는 업무의 5가지 조건을 안다.
  • 단일 에이전트와 멀티 에이전트를 구분한다.
  • 뉴스 동향 분석 미니 에이전트 설계서를 만든다.
  • Claude Code에서 Telegram Bot 발송 흐름까지 연결한다.
도입 질문

지난 1주일간 우리 산업 뉴스를 정리해줘

이 한 줄을 그대로 던지면, 출처 불명 요약과 일반론이 섞일 가능성이 높습니다.
  • 오늘의 핵심은 프롬프트 문장 자체가 아닙니다.
  • 업무 조건, 제외 기준, 출력 구조, 검수 기준을 설계하는 것입니다.
1부

AI는 어떻게 답을 만드는가

AI가 똑똑해 보이는 이유와, 동시에 이상한 답을 내는 이유를 먼저 이해합니다.

1.1

AI는 이해하지 않는다, 예측한다

LLM의 기본 동작

  • 다음에 올 가능성이 높은 단어를 예측합니다.
  • 사용자 질문, 앞뒤 문맥, 학습 데이터의 패턴을 참고합니다.
  • 정답을 안다기보다 그럴듯한 방향을 찾습니다.

한 줄 정리

  • AI의 답변 품질은 "얼마나 똑똑한가"보다 "어떤 패턴을 보게 만들었는가"로 결정됩니다.
1.2

질문이 좁아질수록 답이 정확해진다

입력 문맥모델이 끌어오는 패턴
뉴스 동향을 정리해줘일반적인 뉴스 요약 형식
반도체 산업 뉴스 동향을 정리해줘산업 보고서 톤, 기업명, 정책 키워드
최근 1주일 반도체 수출 규제를 발주사 리스크 관점에서 정리해줘리스크 항목, 영향도, 기간, 대응 방안

좁은 경기장을 만들어주는 것이 AI에게 일을 잘 시키는 첫걸음입니다.

1.3

하나의 채팅창을 계속 쓰면 생기는 일

  • 처음에 준 지시를 잊어버립니다.
  • 중간에 했던 농담이나 톤이 계속 묻어 나옵니다.
  • 다른 주제의 맥락이 섞여 핵심을 놓칩니다.
  • 맥락이 길수록 좋은 것이 아니라, 핵심 신호가 흐려질 수 있습니다.
1.4

Scaling Law와 Inverse Scaling

개념의미체감하는 순간
Scaling Law자원을 더 넣으면 성능이 좋아지는 경향더 좋은 모델로 바꾸면 답이 좋아짐
Inverse Scaling커질수록 특정 과제에서 오히려 틀리는 현상좋은 모델인데 엉뚱한 답이 나옴
Overthinking오래 생각하다가 핵심을 놓치는 현상길게 사고하라고 했는데 더 헷갈림
Context Rot맥락이 길어질수록 초반 지시가 흐려지는 현상채팅창이 길어지면 처음 지시를 무시함
1.5

왜 길어지면 흐려지는가

증상이면의 메커니즘
처음 지시를 잊음최근 발화의 가중치가 커지며 초기 지시가 묻힘
농담 톤이 계속 묻음모델은 톤도 패턴으로 따라감
핵심을 놓침주변 정보가 많아져 중요한 신호의 비중이 줄어듦
잘못된 예시 반복한 번 등장한 예시를 규칙처럼 일반화

오래 생각하게 하는 것보다, 올바른 방향으로 생각하게 만드는 것이 중요합니다.

1.6

10평 집 vs 100평 집

10평 집에서 물건 하나를 찾는 것과 100평 집에서 물건 하나를 찾는 것, 어느 쪽이 빠를까요?
  • AI도 마찬가지입니다.
  • 정보가 많을수록 좋은 것이 아니라, 필요한 정보가 구조화되어 있어야 합니다.
  • 프롬프트는 길이보다 구조입니다.
2부

어떤 업무를 잘 시켜야 할까

AI에게 맡길 수 있는 업무와 맡기면 흔들리는 업무를 구분합니다.

2.1

AI에게 잘 맡길 수 있는 업무의 5가지 조건

  • 목표가 명확하다.
  • 입력 자료가 있다.
  • 결과 형식이 정해져 있다.
  • 판단 기준이 있다.
  • 반복 가능한 절차가 있다.

목표도, 기준도, 결과 형식도 없으면 AI는 그럴듯한 문장을 만들 뿐입니다.

2.2

업무를 작게 쪼개야 한다

뉴스 분석 스킬너무 넓음. 결과가 매번 달라짐
산업 동향 모니터링월별 트렌드 추적
경쟁사 움직임 추적특정 기업 모니터링
규제/정책 변화 분석법/정책 리스크 대응
주간 리스크 브리핑단기 의사결정 지원

좁은 경기장을 여러 개 만드는 것이 넓은 경기장 하나보다 낫습니다.

2.3

좋은 스킬을 구성하는 6요소

요소설명

입력사용자가 제공해야 하는 자료
처리 절차어떤 순서로 판단하고 작성할지

출력 형식최종 결과물의 구조
3부

뉴스 동향 분석 스킬 만들기

네이버 뉴스 기반 주간 산업 동향 분석 시스템을 설계합니다.

3.1

케이스 소개

우리가 만들 것

  • 네이버 뉴스 기반 주간 산업 동향 분석 시스템
  • 입력: 키워드 + 기간
  • 수집: Playwright MCP로 네이버 뉴스 검색 결과 수집
  • 출력: HTML 브리핑 + 텔레그램 그룹 공유

만드는 순서

  • 좋은/나쁜 패턴 정리
  • 출력 구조 설계
  • 처리 절차 설계
  • 검수 기준 만들기
3.2

네이버 뉴스에서 채택할 좋은 패턴

좋은 패턴판단 기준
동일 이슈가 여러 매체에서 반복 보도됨서로 다른 매체 2개 이상
발행일, 기업명, 정책명, 수치가 명확함후속 확인 가능
시장/정책/투자/공급망 영향이 있음단순 제품 소개가 아님
기업 외 이해관계자 시각이 있음정부, 고객사, 경쟁사, 전문가 관점
전후 맥락이 보임발표 -> 반응 -> 영향
3.2

네이버 뉴스에서 피해야 할 패턴

피해야 할 패턴왜 제외하는가
단일 회사 보도자료를 거의 그대로 옮긴 기사의사결정 신호보다 홍보에 가까움
업계 최초, 혁신, 글로벌 리더 같은 문구 중심검증 가능한 근거가 약함
수상, 협약, 출시, 참가만 있는 기사산업 영향이 약함
같은 문장이 여러 매체에 반복 게재반복 보도자료일 가능성
제목만 키워드와 관련 있고 본문은 다른 주제검색 결과 노이즈
3.2

홍보성 기사 판단 기준

회사명 + 제품명 + 수상/협약/출시/선정/참가/프로모션 중심인가?
  • 외부 이해관계자나 시장 영향 분석이 없는가?
  • 여러 매체에 같은 문장으로 반복되는가?
  • 수치나 근거보다 형용사가 많은가?
  • 2개 이상 해당하면 핵심 이슈 후보에서 제외하거나 홍보성 기사로 표시합니다.
3.3

사람이 에이전트에게 전달하는 내용

목표:
최근 7일간 AI 반도체 관련 네이버 뉴스를 수집해서 주간 산업 동향 브리핑을 만든다.

키워드:
AI 반도체, HBM, 파운드리, 엔비디아, 삼성전자, SK하이닉스

대상 독자:
핀테크/IT 서비스를 기획하는 PM 수강생

제외 조건:
단순 회사 홍보성 기사, 반복 보도자료, 광고/스폰서 기사, 기간 외 기사는 제외한다.

출력 형식:
[헤드라인 3줄] [이슈 Top 3] [팩트 타임라인] [상반된 시각] [시사점]
3.3

구조적으로 어떻게 흘러가는가

사람의 요청키워드, 기간, 독자, 제외 조건, 출력 형식, 검수 기준 전달
Collector네이버 뉴스 검색 결과 수집, 메타데이터 저장, 홍보성 기사 제거
Analyst사건 단위 그룹화, Top 3 선정, 사실/해석 분리
Reporter5블록 HTML 브리핑 작성, 텔레그램 메시지 압축
검수출처 수, 날짜, 홍보성 기사 포함 여부, 빈 문장 여부 점검
3.3

결과는 어떤 모습으로 나오는가

[헤드라인 3줄]
1. HBM 공급 경쟁이 이번 주 AI 반도체 뉴스의 중심으로 부상
2. 주요 기업의 증설 발표는 많지만 공급 안정성은 아직 불확실
3. 단순 홍보성 기사는 제외하고 투자/공급망/정책 이슈 중심으로 정리

[이슈 Top 3]
이슈 1. HBM 생산 확대 경쟁
- 사실: A사와 B사가 생산 확대 계획을 발표
- 해석: 2026년 공급 계약 선점 경쟁으로 볼 수 있음
- 출처: 매체 2개 이상

[상반된 시각]
긍정: AI 인프라 병목 완화 가능
우려: 양산 수율과 고객사 확보 여부는 아직 불확실
3.4

검수 기준은 업데이트 장치다

스킬 세팅 전어떤 결과를 좋은 결과로 볼지 기준을 정한다
실행 중에이전트가 결과 제출 전에 기준으로 자가 점검한다
실행 후사람이 결과를 보고 빠진 기준을 체크리스트에 추가한다
다음 실행업데이트된 기준으로 더 안정적인 결과가 나온다
3.4

자가 점검 체크리스트

검수 항목왜 필요한가
핵심 이슈에 출처가 2개 이상 있는가단일 출처 기반 단정을 막기 위해
최근 7일 이내 뉴스만 포함했는가오래된 뉴스를 최신 이슈처럼 쓰는 문제 방지
홍보성 기사를 핵심 이슈에서 제외했는가보도자료가 Top 3로 올라오는 문제 방지
빈 표현이 없는가근거 없는 일반론을 줄이기 위해
사실과 해석이 구분되는가의견이 사실처럼 보이는 문제 방지
원문 전문 전재 없이 요약/인용/링크 중심인가저작권과 출처 문제를 줄이기 위해

4.5

좋은 지시 vs 나쁜 지시

단계애매한 지시더 좋은 지시
수집뉴스 좀 정리해줘최근 7일간 AI 반도체 키워드로 네이버 뉴스 검색 결과를 수집하고 메타데이터를 저장해줘

발행팀에 공유하게 만들어줘텔레그램 메시지 2000자 이하로 압축하고 가장 중요한 이슈 1개를 맨 위로 올려줘
5부

단일 에이전트 vs 멀티 에이전트

처음부터 멀티로 갈 필요는 없습니다. 복잡도가 정당화될 때 역할을 나눕니다.

  • 작성 에이전트
  • 검수 에이전트

핵심

  • 각 에이전트는 자기 일에 특화된 지시와 도구만 받습니다.
  • 오케스트레이터가 결과를 모아 최종 산출물을 만듭니다.
5.4

같은 업무, 두 방식 비교

항목단일 에이전트멀티 에이전트
설계 난이도낮음높음
비용낮음높음
디버깅쉬움어려움

5.5

언제 단일, 언제 멀티?

신호추천 방식

각 단계가 전혀 다른 전문성멀티

시제품/탐색 단계단일 -> 검증 후 멀티
5.6

멀티 에이전트의 함정

  • 에이전트 간 컨텍스트 손실: A가 알아낸 정보를 B가 모름
  • 책임 회피: 검수 에이전트가 문제 없음이라고만 답함
  • 무한 루프: 작성 -> 검수 -> 수정이 끝나지 않음
  • 비용 폭증: 호출 수가 크게 늘어남
  • 디버깅 어려움: 어느 판단이 문제였는지 추적하기 어려움
5.7

하네스 엔지니어링의 등장

멀티 에이전트 문제하네스가 해결하는 방식
컨텍스트 손실단계별 입출력 형식을 고정하고 필요한 정보만 넘김
책임 회피각 에이전트의 책임, 허용 도구, 금지 도구 명시
무한 루프재시도 횟수, 종료 조건, 사람 승인 지점 설정
비용 폭증단순 작업은 작은 모델, 어려운 판단은 큰 모델로 라우팅
디버깅 어려움각 단계의 입력, 출력, 판단 근거를 로그로 남김

하네스는 AI가 일을 망치지 않도록 작업 환경을 설계하는 기술입니다.

5.8

Hermes는 무엇을 보여주는가

설계 요소의미
Main model주된 대화와 핵심 판단을 담당하는 모델
Auxiliary models요약, 검색, 압축, 승인 판단 같은 보조 작업을 맡는 모델
Tool routing어떤 도구를 언제 호출할지 정하는 구조
Memory / Context이전 작업과 사용자 선호를 보존하는 구조
Sub-agent / Skills반복 업무를 별도 역할이나 스킬로 분리

모델을 잘 고르는 것보다, 모델이 일하는 환경을 잘 설계하는 것이 중요해지고 있습니다.

6부

AI를 잘 쓰는 작은 습관

프롬프트보다 중요한 것은 단어, 채팅 분리, 모델 분배 습관입니다.

6.1

단어를 알면 한 마디면 된다

하고 싶은 말알아두면 좋은 단어
많은 자료에서 의미 있는 패턴을 찾아줘데이터 마이닝
문서를 잘게 나눠 검색 가능하게 만들어줘청킹, 임베딩
검색한 근거를 바탕으로 답변하게 해줘RAG
역할 분담된 여러 AI가 협업하게 해줘Multi-Agent, Orchestration
여러 모델을 작업별로 분배해서 쓰게 해줘Model Routing, Harness
AI가 브라우저/외부 도구를 직접 다루게 해줘MCP
6.2

메인 채팅 vs 임시 채팅

메인 채팅

  • 프로젝트 진행
  • 결정
  • 자료
  • 요구사항

임시 채팅

  • 모르는 용어 학습
  • 실험
  • 테스트
  • 기초 질문
6.3

작업마다 다른 모델 쓰기

작업 유형추천 모델 특성이유
검색/수집/요약작고 빠른 모델단순 작업, 속도 중요
깊은 분석/판단크고 정교한 모델다단계 추론 필요

형식 변환/정제작은 모델패턴이 명확
최종 검수큰 모델누락/오류 탐지
7부

실습: 멀티 에이전트 설계하기

네이버 뉴스를 매주 자동 수집/분석해서 텔레그램 방으로 보내는 시스템을 설계합니다.

7.1

실습 설정

항목설정
처리 대상네이버 뉴스 검색 결과
수집 방식Playwright MCP 기반 브라우저 자동화
키워드수업에서 합의
기간최근 7일
출력물HTML 5블록 보고서 + 텔레그램 메시지

7.2

실습 진행 단계

  • 자동화하고 싶은 반복 업무를 한 줄로 좁힌다.
  • 좋은/나쁜 결과 패턴을 정리한다.
  • 업무를 3개 sub-agent로 분해한다.
  • 각 sub-agent의 입력/출력/도구 권한을 명시한다.
  • Orchestrator 호출 순서와 중간 검증 포인트를 작성한다.
  • 최종 출력 형식과 발행 채널을 정한다.

7.6

Collector 핵심 설계

이름: news-collector
책임: 네이버 뉴스 검색 결과를 수집해 기사 메타데이터와 요약을 저장한다.
허용 도구: Playwright MCP, Write
금지: 원문 전문 저장, Edit, Delete
절차: robots.txt/약관 확인 -> 뉴스 탭 검색 -> 메타데이터 수집 -> 중복/홍보성/기간 외 제거 -> 요약/인용/링크 저장
7.7

Analyst 핵심 설계

이름: news-analyst
책임: 수집 파일을 읽고 핵심 이슈와 상반된 시각을 추출한다.
허용 도구: Read, Grep, Glob
금지: WebSearch, WebFetch, Write
절차: 파일 목록 확인 -> 사건 단위 그룹화 -> 사실/해석 분리 -> Top 3 선정 -> 상반된 시각 탐색
7.8

Reporter 핵심 설계

이름: news-reporter
책임: 분석 결과를 5블록 HTML 보고서로 작성하고 텔레그램으로 발송한다.
허용 도구: Read, Write, Bash
금지: WebSearch, WebFetch
절차: 5블록 작성 -> 자가 점검 -> 텔레그램 메시지 압축 -> 환경변수 확인 -> sendMessage 호출 -> ok:true 확인
7.9

Orchestrator 지시문 예시

이번 주 AI 반도체 동향 브리핑을 만든다.

1. news-collector 호출
   키워드: AI 반도체, HBM, 파운드리
   기간: 최근 7일

2. 수집 결과 확인 후 news-analyst 호출
   관점: 발주사 리스크

3. 분석 결과 확인 후 news-reporter 호출
   채널: 텔레그램 그룹

4. 각 단계 사이에 결과 요약을 보여주고 실패 지점을 명시한다.

8부

정리

AI에게 일을 잘 시키는 사람은 프롬프트를 길게 쓰는 사람이 아니라, 업무 구조를 아는 사람입니다.

8.1

오늘의 핵심 6가지

  • AI는 패턴 예측 모델이다.
  • 오래 생각한다고 항상 좋아지는 것은 아니다.
  • 프롬프트는 길이보다 구조다.
  • 좋은 업무는 작게 쪼갤 수 있다.
  • 좋은 에이전트는 좋은 업무 설계에서 나온다.
  • 단일로 시작해 검증하고, 필요할 때 멀티로 분리한다.
